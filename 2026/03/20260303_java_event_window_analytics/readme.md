# 今日のJava課題：**Event Window Analytics（時間窓つきイベント集計）**

## 問題
イベントログ（時刻つきの発生記録）がストリーミングで与えられます。  
あなたは「直近W秒の窓（スライディングウィンドウ）」の中だけを対象に、集計・問い合わせに答えるプログラムを作ってください。

ログは時刻 `t`（整数、秒）で到着し、**入力は時刻が非減少（同時刻はあり）**で並びます。

---

## 入力形式
最初の1行：
- `Q W`（クエリ数Q、窓幅W秒）

続くQ行は次のいずれか：

1) イベント追加  
- `ADD t type user`
  - `t`：時刻（整数）
  - `type`：イベント種別（英小文字/英大文字/数字/`_` のみ、長さ1〜20）
  - `user`：ユーザID（同様の文字種、長さ1〜20）

2) 直近W秒に含まれるイベント総数  
- `COUNT t`

3) 直近W秒に含まれる **ユニークユーザ数**  
- `USERS t`

4) 直近W秒における **typeごとの回数の最大値**（最頻typeの回数）  
- `MODE t`

5) 直近W秒における **指定typeの回数**  
- `TYPE t type`

---

## 窓の定義（超重要）
問い合わせ時刻を `t` とすると、対象となるのは

- **`(t - W, t]`** の範囲にあるイベント

つまり、
- `t` は含む
- `t - W` **ちょうど**のイベントは含まない（開区間）
- 例：W=10, t=20 の窓は `(10, 20]` → 11〜20 のみ

---

## 出力形式
問い合わせ（`COUNT/USERS/MODE/TYPE`）のたびに、その答えを1行で出力してください。

---

## 制約
- `1 ≤ Q ≤ 200000`
- `1 ≤ W ≤ 10^9`
- `0 ≤ t ≤ 10^9`
- 入力は時刻が非減少（`ADD` と問い合わせの `t` も含め全体で非減少）
- メモリは常識的な範囲（全ログ保持は不可な想定）

---

## 例
### 入力
```
12 5
ADD 1 click u1
ADD 2 click u2
ADD 4 view u1
COUNT 4
USERS 4
MODE 4
TYPE 4 click
ADD 7 click u1
COUNT 7
USERS 7
MODE 7
TYPE 7 view
```

### 出力
```
3
2
2
2
2
1
1
1
```


---

## 要求される実装方針（ヒント）
- 窓から外れたイベントを素早く捨てる必要があります  
  → **キュー（Deque）**で「窓内イベント」を保持し、問い合わせ時刻ごとに古いものをpopして更新
- 集計は以下を持つと良いです：
  - `type -> count` のMap
  - `user -> count` のMap（窓内にそのユーザが何件いるか）
  - `現在のユニークユーザ数`
  - `typeの最大頻度（MODE用）` を更新する仕組み（ここが一番ムズい）

---

## 追加の設計メモ（MODE攻略の方向性）
`MODE`（typeの最大回数）を高速に答えるには、例えば：
- `typeCount: Map<String, Integer>`
- `freqCount: Map<Integer, Integer>`（頻度fのtypeが何種類あるか）
- `currentMaxFreq`（現在の最大頻度）
を持ち、イベント追加/削除で
- typeの頻度が `a -> a+1` なら `freqCount[a]--`, `freqCount[a+1]++`
- 最大頻度が空になったら `currentMaxFreq` を下げる
のように管理すると、ならしで高速になります。

---

## ゴール
- `O(Q log M)` または **ならし `O(Q)`** を狙える設計にする（Mはtype/userの種類数）
- 少なくとも `Q=200000` を余裕で通せる実装にしてください

---

## 提出物
- `Main.java` 1ファイルでOK（標準入力/標準出力）
- 速度のため `BufferedInputStream` + 自作高速パーサ推奨
