# 今日のC++課題：Sliding Window Analytics（イベント集計エンジン）

## 概要
リアルタイムに流れてくるイベントログを処理し、**直近W秒のスライディングウィンドウ**内での集計結果を返すミニ分析エンジンをC++で実装してください。

- 入力は時刻が単調増加（同時刻はあり得る）するログコマンド列
- ウィンドウは「最新時刻Tに対して **(T - W, T]** に入るイベントだけを有効」とする（※T-Wちょうどは除外）
- 期限切れイベントを素早く捨てながら、集計に即応できる構造を作るのがポイント

## 入力形式
1行目：
- `Q W`  
  - `Q` : コマンド数（1 <= Q <= 200000）
  - `W` : ウィンドウ幅（秒）（1 <= W <= 10^9）

2行目以降：Q行のコマンド
- `ADD t user type`
  - 時刻 `t`（0 <= t <= 10^9）
  - ユーザID `user`（英小文字/数字/アンダースコア、長さ1〜32）
  - イベント種別 `type`（同上）
- `COUNT t`
  - 時刻 `t` におけるウィンドウ内イベント総数を出力
- `TOPUSER t k`
  - 時刻 `t` におけるウィンドウ内でのユーザ別イベント数の上位k件を出力
- `TOPTYPE t k`
  - 時刻 `t` におけるウィンドウ内での種別別イベント数の上位k件を出力

## 出力形式
- `COUNT`：1行に整数
- `TOPUSER` / `TOPTYPE`：k行出力（ただし実在する件数がk未満ならある分だけ）
  - 形式：`<name> <count>`
  - 並び順：
    1. count 降順
    2. name 昇順（同率のタイブレーク）

## 重要ルール（ここが罠）
- `ADD` と `COUNT/TOP...` の `t` は **その時点の“現在時刻”** を意味する  
  → つまり `COUNT 100` が来たら、時刻100として期限切れを捨ててから答える
- 入力の時刻は **単調非減少**（前の行より小さくならない）とする  
  → これを使って O(Q log Q) 以内を狙える

## 制約と期待計算量
- Qが最大20万なので、`TOP...`を毎回全ソートすると死にます。
- 目標：平均的に **O((Q + 出力件数) log N)** 程度

## 例
### 入力
```
12 5
ADD 1 alice click
ADD 2 bob click
ADD 4 alice view
COUNT 4
ADD 7 alice click
COUNT 7
TOPUSER 7 3
TOPTYPE 7 3
COUNT 10
TOPUSER 10 10
ADD 10 bob view
COUNT 10
```

### 出力
```
3
3
alice 2
bob 1
click 2
view 1
1
alice 1
1
```


（例の解釈：時刻7・W=5ならウィンドウは(2,7]なので、t=2は除外、t=4/7が残る…など）

## 実装要件（推奨）
以下の方針のどれか（または組み合わせ）で実装してください。

### 必須
- 期限切れイベントを捨てるために、イベントを時系列で保持する（例：`deque`）
- 集計のためにカウントを保持する（例：`unordered_map<string,int>`）
- `COUNT` は O(1) で返せるようにする

### TOPを速くするヒント（どれを選んでもOK）
1) **遅延削除つきヒープ**
- `priority_queue` に (count, name) を積む
- 更新時に新しいペアを積み、取り出すときに map と一致しない古い要素を捨てる

2) **bucket/ordered set**
- `map<int, set<string>>` みたいに「count→名前集合」を持つ
- countの増減に応じて集合を移動させる（上位を取りやすい）

3) **policy-based data structure（使える環境なら）**
- GNU拡張だが競プロならアリ（今回は必須ではない）

- スライディングウィンドウ（dequeでの期限切れ処理）
- 集計（ハッシュマップ）
- “上位K” クエリを、全ソートせずに捌く設計
- 遅延削除（lazy deletion）の考え方

## 追加チャレンジ（余裕があれば）
- `TOPUSER` と `TOPTYPE` を同じ枠組みで扱えるようにクラス設計する（テンプレでもOK）
- `ADD` のイベントに「重み w」を追加して、カウントを加算にする
- メモリを抑える（短命な文字列の扱い：string interning等）
